Issue #539: Batched building for test cases
------------------------------------------------------------
For a test case compilation time can take quite a bit of time compared to the execution of the test. Given we test our llk separately from tt-metal, for each test case we have fewer files to test, that would mean we are probably not using the full capacity of the machine while compiling, even if we parallelize the compilation of the files of a single test case.  So we could parallelize the compilation more, in a producer consumer fashion. 

For example multiple CPU threads will try to compile multiple test cases into a folder as long as there is space in the folder and there are test cases to be executed. 

While the execution thread will keep on executing tests as long as compiled test cases are finished. 

If compilation time takes considerable time compared to execution time, this should speed up the testing considerably.

Issue #523: Profiling and analysis of Reader/Writer/Compute Kernels for an OP
------------------------------------------------------------
If we consider the three kernels as three stages of a pipeline, the one that will end up being the bottleneck is going to be the one with the largest stage. In an ideal scenario, these three would take exactly same amount of time and none would bottleneck another.

In a practical scenario this is not achieved and one stage waits for another, now how is that wait manifested ?

It is manifested through the wait times for a circular buffer for the second iteration and so on. In the first iteration, the compute kernel must wait for the circular buffer to be filled, the writer must wait for its corresponding circular buffers to be filled, but theoretically in an ideal case the moment the compute kernel tries to read new data from the circular buffer, it should be there. 

So one thing we could do is analyze which buffers are being waited upon, that could indicate who is causing the bottleneck, but then again due to dependency chain it might not be obvious and may require some investigation. For example the writer waits for the compute unit to finish filling up the circular buffer, but that could just be because the compute was slowed down by the reader. 

So how do we know who is the bottleneck ? Here is a plausible way of doing the analysis. 

Let the reader kernel run with an infinitely fast compute kernel & writer kernel (that is assume data is consumed right away) 
Let the compute kernel run with an infinitely fast reader & writer kernel (that is work with garbage data , assuming the compute speed remains the same). 
Let the writer kernel run with an infinitely fast reader & compute kernel. 

This should tell us which kernel is taking up the most time and needs to be optimized, if the reader and writer kernels are taking more time, then we should probably change the data format. 


But here is the catch. In some cases in this train of thought there might be more hypothetical pipeline stages. That is when the reader kernel is broken into two parts & so is the compute & writer kernels. In such cases its like a many stage pipeline, and we need to find out a sorted list latencies of such stages and spot the bottleneck. I think we could still do that by applying the above methods, we just need to use timestamp for each such stage. The idea would be the same, benchmark each part in a thread assuming everything else is running at full speed. And then make a sorted list of portions of work based on time consumed and then we could focus on improving those chunks. 

It may be possible to reverse engineer the bottlenecks by looking at the waiting time on the circular buffers (which are the effects, the differences in processing time of the chunks being the cause), but that would require a bit of mathematical analysis and thought. Or it may be just measuring everything in chunks and ignoring the wait times and then creating a sorted list of some form.

Issue #494: SFPU bfloat16 binary LLK truncate towards 0 rather than round-to-nearest
------------------------------------------------------------
## Problem

To give some context: we have accuracy issues with some operation. In particular, the pow operation gives `9**2 ~78` for bfloat16, instead of 81.

Fortunately, we have found a better approximation for `pow`. 
This [new approximation](https://github.com/tenstorrent/tt-llk/blob/5d8b851f70679467821835760416bbea7037f137/tt_llk_wormhole_b0/common/inc/sfpu/ckernel_sfpu_binary.h#L82) gives us `9**2 = 80.5`, which is 1 ULP away from the truth (i.e. there are no other bfloat16 numbers between 80.5 and 81).

Interestingly enough, the float32 operation gives `9**2 = 80.819`. 

If I understand correctly, the SFPU does all its internal computation in float32.  If that's indeed the case, then the final `bfloat16` value is rounded from the `float32` upon being written (to a DEST register ?).

The general expectation is that `80.819` should be rounded up to `81` rather than truncated down to `80.5`, which is not the case here. 


## Reproduction steps

The following tests have been performed on wormhole. 

For testing, I am currently using the [nmaurice/test-binary-llk](https://github.com/tenstorrent/tt-metal/tree/nmaurice/test-binary-llk) branch of tt-metal, and the [nmaurice/exp-accurate](https://github.com/tenstorrent/tt-llk/tree/nmaurice/exp-accurate) branch of tt-llk (which has the new power function). 
The test program/kernels is at [tt_metal/programming_examples/test-binary-llk](https://github.com/tenstorrent/tt-metal/tree/nmaurice/test-binary-llk/tt_metal/programming_examples/test-binary-llk).

It can be compiled and run with 


```
CONFIG=Debug ./build_metal.sh --build-tests -b Debug -e --build-programming-examples
./build_Debug/programming_examples/metal_example_test_binary_llk
```

This will perform `pow(input0, input1)` for several input values and print results.

In this case, the compute kernel also prints the computed tile with 'dprint_tensix_dest_reg()'. With bfloat16, it shows a DEST tile filled with 80.5 (instead of 81).

Manually changing calls to `_calculate_sfpu_binary_power_` to `_calculate_sfpu_binary_power_null_`, which only returns `80.819f` gives the same result.

## Attempted solution


For bfloat16, the `float_to_fp16b(x, 0)` intrinsic can be called, and solves the problem. However, it would also massively degrade accuracy for float32. And I'm not exactly sure how to pass the output data format to the LLK to enable/disable it.

Issue #488: Test test-cases with reordering of functions/instructions.
------------------------------------------------------------
We have many test cases where some functions could be reordered, and those test cases would only fail with certain orders. 

For example the following test case in issue : https://github.com/tenstorrent/tt-metal/issues/22068

```
void MAIN {
    // Circular Buffers
    constexpr auto cb_in = tt::CBIndex::c_0;
    constexpr auto cb_other = tt::CBIndex::c_1;
    constexpr auto cb_out = tt::CBIndex::c_2;

    // Variables
    constexpr uint32_t input_dst_reg = 0;

    add_trisc_nops<UNOPS, MNOPS, PNOPS>();

    binary_op_init_common(cb_in, cb_other, cb_out);

    tile_regs_acquire();
    cb_wait_front(cb_in, onetile);
    cb_wait_front(cb_other, onetile);

    mul_tiles_init(cb_in, cb_other);
    mul_tiles(cb_in, cb_other, 0, 0, 0);

    tile_regs_commit();

    tile_regs_wait();
    cb_reserve_back(cb_out, onetile);
    pack_tile(0, cb_out);
    cb_push_back(cb_out, onetile);
    tile_regs_release();

    cb_wait_front(cb_out, 1);
    cb_pop_front(cb_in, onetile);
    cb_pop_front(cb_other, onetile);
}
```

This test case would only fail if cb_wait_fronts are put after the binary_op_init_common, not before. The details can be found in the issue (there is a race between the packer and unpacker initializations). Such cases are generally race conditions, and they take a huge amount of resources from different team during debugging. 

This race condition can be reproduced frequently by adding perturbation before the binary_op_init_common, but the prerequisite was having the call to cb_wait_front after binary_op_init_common. 

Now to have such reproducers, one thing we could do is in a kernel set some ordering among instructions through dependencies, and then we could have a program generate the test case everytime it tests in a different order along with other features like adding perturbation/randomly initializing the state before starting the kernel to maximize chances of encountering the errors. 

But for such a test case before running it must record the ordering so that we can get back to it if we need to debug it.

Issue #482: Test for races in updating initial values of registers.
------------------------------------------------------------
When a kernel is run and it sets the values of a register and a subsequent kernel is run that needs the values of the registers to be reversed, there can be race conditions leading to wrong outputs when two threads of the next kernel tries to update the same register. 

An example is this issue : https://github.com/tenstorrent/tt-metal/issues/22068

See comment : https://github.com/tenstorrent/tt-metal/issues/22068#issuecomment-3045684363

The test case runs a copy tile kernel and then an elementwise multiply kernel. 
The issue lies in the race on the ALU_ACC_CTRL_Zero_Flag_disabled_src bit (datacopy needs this bit set while mul needs it cleared), in the failing iteration this bit fails to clear (gets cleared for only a couple of cycles and then restored) causing wrong FPU results. The cause of the race is that the register that stores that bit also stores some packer bits meaning that the same register is accessed from both the unpacker and the packer thread

Unpacker thread uses tensix RMW instructions to modify that bit (cunpack_common.h:294) 

```
   template<bool row_pool=false, bool is_fp32_dest_acc_en = false, bool fpu_srnd_en = false, bool pack_srnd_en = false>

      uint32_t src_zeroflags_disable = ((uint)unpA_dst_format == (uint)DataFormat::UInt16) || ((uint)unpB_dst_format == (uint)DataFormat::UInt16);
      cfg_reg_rmw_tensix<ALU_ACC_CTRL_Zero_Flag_disabled_src_RMW>(src_zeroflags_disable);
```

while packer uses explicit risc mmio read modify write sequence outside of the appropriate mutex guards (cpack_common.h:554) 

```
@@ -435,16 +443,6 @@ inline void configure_pack(
    regfile[p_gpr_pack::TILE_HEADER + 3] = 0;
    sync_regfile_write(p_gpr_pack::TILE_HEADER + 3);

    relu_config_u hw_relu_config;
    // Config RELU
    uint32_t current_relu_val = reg_read((uint)&cfg[STACC_RELU_ApplyRelu_ADDR32]);
    hw_relu_config.val[0]     = current_relu_val;

    hw_relu_config.r.STACC_RELU_ApplyRelu     = relu_config & 0xffff;
    hw_relu_config.r.STACC_RELU_ReluThreshold = (relu_config >> 16) & 0xffff;

    cfg[STACC_RELU_ApplyRelu_ADDR32] = hw_relu_config.val[0];
```

if the unpack RMW happens between the pack read and write the unpack change will be lost.


Action items could include :

--- Look at all MMIOs which write to more bits than needed and replace them with cfg_reg_rmw or check if there are potential race conditions and use synchronization mechanisms. 

--- For every test case we test it in two ways, we set all the registers to true/false in the beginning and start the test case with some perturbation.
 
For example this particular issue mentioned here https://github.com/tenstorrent/tt-metal/issues/22068 was not easily reproducible, it was made reproducible by adding perturbation to it, at the very beginning before the initialization functions. The branch and commands are mentioned in the issue description.

Issue #406: [Test infra] Inability to Reliably Read Integer Data from dest Registers on Wormhole
------------------------------------------------------------
In order to do integer format sweep we need to be able to have visibility into device.
We’re currently unable to reliably read integer data from dest registers on the Wormhole architecture due to a halting bug that prevents us from holding the core in place for memory reads. 
On Blackhole, we are able to use Exalens to read dest registers and extract integer data using the standard integer formats. However, on Wormhole, there is a hardware-level bug that prevents Exalens from halting the core, making memory reads unreliable or impossible in some cases.

To unblock myself from waiting on Exalens to support integer decoding on Wormhole, I attempted a workaround:
- Dump raw hex data from srcA and dest registers using tt-exalens functions that will read bytes regardless of format.
- Manually parse this data back into the intended format 

While this approach technically works in some cases, it suffers from several limitations on Wormhole:
- MOVDBGA2D instruction (used by Exalens) truncates the lower 3 bits of the integer when copying to dest, resulting in loss of precision. After consulting with Marco Merlini he confirmed that this may be one potential bug. Small int8/uint8 values are particularly affected — if the magnitude is too small, it ends up in the truncated bits and appears as zero.
- Because of this truncation, there's very little flexibility or variety in testing values — most low-range values are wiped out, and data integrity is compromised.

There also appears to be a deeper hardware bug that affects the ability to accurately read or preserve lower-bit data on Wormhole. I consulted with Marco Merlini, who noted that the issue is likely rooted in the RTL, but it's complex and currently not well understood or easily traceable.

So we will wait on debug team to provide integer support for reading integer values before moving forward

Issue #397: Investigate underlying context/state leakage after int32 permute operations in LLK
------------------------------------------------------------
Recent debugging revealed that running int32 permute tests can leave ttnn in an unexpected state, affecting subsequent operations (e.g., bfloat16) due to insufficient reset of internal context, config, or state. While a workaround/fix has been identified and applied, the root cause of the state leakage or improper reset remains unclear.

Issue #339: [ Feature ] higher-precision arithmetic emulation in the matmul unit
------------------------------------------------------------
Right now the highest precision the matmul unit supports on any tenstorrent card is TF32, and as far as I can tell the effective machine epsilon is around 2^-9 or even lower. This is too high for almost any numerical linear algebra workload and even a lot of machine learning workloads. It is possible of course to do everything in the SFPU and use full precision arithmetic, but this seems very nontrivial and probably slow due to the lack of cross-lane addition instructions.

The simplest strategy to do this I think would be to create some new kernels which "unpack" a 16x16 matrix of fp32 values into a 32x16 matrix in srcA or a 16x32 matrix in srcB, both of TF32 values. Each 16x16 block of these matrices just stores one part of the original numbers from the input matrix, so for instance if unpacking a matrix X into srcB, we would store [X1, 2^-10\*X2] where X = X1 + 2^-10\*X2. The "unpacking" procedure that the SFPU needs to do would basically just just be VD = VC - Truncate(VC), where the truncation is either to the high 19 bits for srcB or the high 18 bits for srcA. Then we'd do an effective 8 fidelity passes for 18 bits of effective mantissa, which is enough for most applications.

This issue is to track discussion relating to an eventual PR that I intend to submit (work starts either this week or next week).

Questions:
-Does this feature properly belong in the llk library?
-If so, is there any special pattern that needs to be adhered to for the kernels to be usable in higher-level software stacks?
-Is it advisable to have the SFPU part of these "unpacking" kernels be dispatched from the actual unpacker thread? I think it would be nice to not need to constantly reconfigure the replay buffer of the math thread, and it doesn't seem like the unpacker thread will have much to do most of the time.

Issue #291: Create a single threaded program for documentation purposes, that uses only one TRISC core to perform the tasks of all cores.
------------------------------------------------------------
For documentation purposes we want to write a program that handles the task of all the cores in a tensix core. The single core will handle all the responsibilities of the Reader/Writer/Unpacker/Math & Packer. This will serve the purpose of explaining how things are done in Tensix Cores. We would move on to explain how we divide the different tasks to the different cores and how. 

Key requirements. 

- Simple and easy to understand :  We should choose a simple program but also one that uses all the cores. A program using a simple eltwise addition will do. 

- Highly documented : Since this is for documentation purposes, we want the code to be highly documented and nicely organized. 


**Steps taken (assuming the TRISC0 will execute all the compute code) :**

- Make copies of api calls (with small modification in name) and replace PACK / MATH with UNPACK as we want TRISC0 to execute all the functions called in the api functions. 

- In the api file used, include math and unpacker code/headers unconditionally since all the code needs to be compiled for the TRISC0/unpacker thread. This will lead to conflicts for duplicate declarations. For example is_32bit_input is implemented for both unpacker and math. Such cases should be handled. (In this case comment out the code for the math thread).

 ```
 chlkc_math.cpp		chlkc_pack.cpp		  chlkc_unpack.cpp	    defines_generated.h
 ```
 for math and packer needs to be accessible to the unpacker. For that we need to modify the ```chlkc_list.h``` file where it conditionally includes for unpacker/math/packer. Two other files (dataflow_api.h and dprint_tile.h) include some of these files but that should not be a problem because they do not include these files based on whether it is the unpacker/math/packer.

The last step did not pan out properly. Files like 

```
tt_metal/include/compute_kernel_api/common_globals.h
/tt_metal/hw/ckernels/blackhole/metal/llk_io/llk_outputs.h
```

Threw errors regarding undeclared identifiers, and I had to include

```
in common_globals.h
```


- The different threads have their own local circular buffers, due to some optimizations all the read write pointers are not initialized. For example for the unpacker's interface the write pointer is not updated. After talking with relevant developers it seems this was done for completely optimization purposes. So all of them can be initialized when using the unpacker thread to do all tasks. Apply this change to trisc.cc

```
-#if defined(UCK_CHLKC_UNPACK)
+// These were conditionally set to true/false
+// do avoid writing, but both being true should
+// work, as read and write initializes read and
+// write pointers respectively.
 constexpr bool cb_init_read = true;
-#else
-constexpr bool cb_init_read = false;
-#endif
-#if defined(UCK_CHLKC_PACK)

-#else
-constexpr bool cb_init_write = false;
-#endif
```

- Merge and remove mops

A thread has access to only one mop buffer, thus we cannot have separate mops anymore and configuring one will overwrite the previous one. Thus we merged the mops of unpacker and math and did the work of packer without a mop. Moreover since we have access to the addressmods of one thread only, we make sure that the function to unpack and do math does not conflict with the addressmods used by the function to pack. This way we can still initialize only once and run the functions / mops repeatedly, since addressmods do not conflict and we run only one mop. 

- Remove register sync functions 

Since only one thread is doing all the compute parts, there is no need to sync destination register using the functions like ```tile_regs_acquire/wait/commit/release```

Issue #290: Review transpose_dest API
------------------------------------------------------------
As part of #250, it was found that `transpose_dest` wasn't that widely used (the only place I could find was for row-wise cumsum) prior to adding support for transposing 32-bit values.

To reduce API surface area, it might be worth revisiting to see if the API can be simplified.  One idea is to drop the 32x32 tile `transpose_dest` entirely in favour of a single API for the aforementioned 16x16 face transpose for 32-bit (and maybe non-32-bit). **Improve the golden generation fidelity to better reflect kernel-level precision behavior**

This points to a limitation in the golden generation logic: 
- Golden does not fully capture the minor numerical differences that accumulate when these ops are executed through the actual kernels, particularly under lower math fidelity settings.

Issue #177: Dealing with files shared between tt-llk and tt-metal
------------------------------------------------------------
Currently, we have several header files that reside in the `tt-metal` repository but are required for the LLK test infrastructure to compile and execute:
1.  `cfg_defines.h`
2. `dev_mem_map.h`
3. `tensix.h`
4. `tensix_types.h`
5. `ckernel_sfpu_converter.h`

However, since they are frequently updated in the `tt-metal` repository, we need to continuously track these changes to keep them in sync. This approach is not sustainable.

Since these files are essential and cannot be removed, an ideal solution would be to store them in a centralized location (e.g., a separate repository) that all other repositories could access. However, this would complicate matters for existing users (e.g., metal, umd).

The proposed solution is to download these files from `tt-metal` dynamically whenever LLK tests are run. This approach is minimally intrusive, as it only affects our repository and does not impact others. The main drawback is that we may encounter sudden CI failures if changes are made to these files in `tt-metal` and we are not prepared for them. Also, later, if we encounter issues with stack/code size, we could come back to the previous approach, as we would need to have our version of the above, suitable for testing. Issue #129: [Test Infra] Inconclusive Hardware Definition for Packer Format Conversion from Float16_b to Float16_a
------------------------------------------------------------
I have a situation where I want my input tensor in L1  to be in format `Float16_b` and my output tensor in L1 to be in format `Float16_a`. I am currently setting my unpacker formats (unpack_src, unpack_dst) to be (`Float16_b`, `Float16_b`) as is supported in hardware. 
However for programming packer formats I am getting conflicting advice from hardware team regarding which formats Gasket can convert and which formats packer can accept as input and corresponding output format it can convert. 

Marko Merleni claims that Packer can take input of Float16_b and convert it to Float16, however when I run this in the test infrastructure the result is wrong, after consulting with Marko he concluded that the number is not being converted to Float16 and reading it as Float16 gives us the wrong results we are getting in output. He suggested that certain registers need to be set to specific values for packer to be able to do the conversion. When using tt-exalens to check the value of those registers some of them are not set to the values Marco specified 
- `PCK_DEST_RD_CTRL_Round_10b_mant` should be 1 instead of 0
- `THCON_SEC0_REG1_Exp_section_size` should be set to 0, in our case it is set to 4
This would require that we go into `pack_common.h` and program registers accordingly for this specific case, but that seems a bit tricky.

On the other hand, Rakesh Lal provided a table for format conversions in blackhole which shows that Packer cannot covert `Float16_b` to `Float16`, which contradicts Marco's table and suggestion.  Rakesh suggests that Packer Gasket does format conversion from `Float16_b` to `Float16`. In which case it would make sense to set Packer input and output formats to be `Float16`, because Gasket will convert `Float16_b` format from dest register to `Float16` accordingly. 
However when I try to program it this way, I get extremely big numbers in my output to L1. Upon further inspection it turns out that these numbers are coming from the dest register which is storing these huge numbers. 
- Interestingly enough the only thing the changed from this case and the first case is how we program packer formats, in the case above I set input and output to be `Float16_b` and `Float16`, int his case they are `Float16` and `Float16`. This leads me to believe that dest register format or numbers are behaving in accordance to how we program packer formats, which is counter intuitive because dest register format depends on which formats unpacker stores into source register and fpu later works with.

Issue #122: Reduce kernel on wormhole [P1]
------------------------------------------------------------
Whilst attempting to complete the [`phi-1` bounty](https://github.com/tenstorrent/tt-metal/issues/18287), we bumped into `layer_norm` having very bad precision.

The lack of precision for `layer_norm` also seems to manifest for other models (e.g https://github.com/tenstorrent/tt-metal/issues/18548)

That led me to the wormhole reduce kernel, which seemingly ignores `is_fp32_dest_acc_en`:  https://github.com/tenstorrent/tt-llk/blob/main/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L168

Accumulating in FP32 is very important for these types of operations, so I'm unsurprised that it currently has bad PCC.

The oracular @pcawleyTT had this insight:

> The code you linked to looks like it was originally written for Grayskull, and then piecemeal ported to Wormhole, and then piecemeal hacked on (hence all the commented out bits). There's a TTI_TRNSPSRCB a few lines down from the line you linked, which poses as problem on Wormhole, as SrcA / SrcB only hold 19b datums, whereas Dst can hold 32b datums, so the Grayskull approach of transposing Dst via moving it to SrcB results in precision loss. is_fp32_dest_acc_en ties in to whether Dst contains 16b datums or 32b datums, but nothing can make SrcB any wider.
The approach it should be taking for 32b datums is to split them in to two 16b parts (or a 19b part and a 13b part), use two SrcB transposes to transpose the parts, and then rejoin the parts.

Issue #115: Suspected compute kernel issue when scalling behond 1 tile
------------------------------------------------------------
Copying in a customer issue:

> 
> To demonstrate the problem, I've implemented a simplified tile addition program that should be a very good representation of why one of the all-reduce algorithms is not working, and hopefully also holds the answer to why another all-reduce algorithm isn't working as expected.
> Overall, the program functions as follows:
> Host: Creates a bfloat_16 vector containing all 1s
> Host: Copies the vector to shared-device memory
> Host: Launches kernels
> d_Dataflow: Copies vector to node memory, in "cb_local" and "cb_recv"
> d_Compute: Adds cb_recv directly to cb_local, iterating over cb_recv but staying on tile 0 of cb_local and writing the results directly to cb_local
> d_Dataflow: Copies vector from node to shared-device memory
> Host: Copies vector from shared-device to host memory
> Host: Compares with output
> The behaviour is as follows:
> Given 0 tiles to add - compute kernel doesn't execute the loop, doing no math - answer correct (no change 1+0=1)
> Given 1 tile to add - compute kernel executes the loop exactly once, adding cb_recv tile 0 to cb_local tile 0 - answer correct (1+1=2)
> Given n > 1 tile to add -  compute kernel executes the loop n times, always adding cb_recv tile i to cb_local tile 0 - answer incorrect (outputs 2 regardless of how many times the addition loop is carried out).
```

namespace NAMESPACE {
void MAIN {
    uint32_t num_tiles = get_arg_val<uint32_t>(0);
    uint32_t cb_wait_tile_num = num_tiles;
    if (cb_wait_tile_num < 1) {
        cb_wait_tile_num = 1;
    }

    constexpr uint32_t cb_id_SE = tt::CBIndex::c_2;
    constexpr uint32_t cb_id_recv = tt::CBIndex::c_3;
    constexpr uint32_t cb_id_local = tt::CBIndex::c_16;

    // await writing to local buffer to finish
    cb_wait_front(cb_id_local, 1);
    cb_wait_front(cb_id_recv, cb_wait_tile_num);

    binary_op_init_common(cb_id_local, cb_id_recv, cb_id_local);
    add_tiles_init();
    //In place add of recv buffer to local buffer
    for (uint32_t n_tile = 0; n_tile < num_tiles; n_tile++) {
        tile_regs_acquire();
        add_tiles(cb_id_local, cb_id_recv, 0, n_tile, 0);
        tile_regs_commit();

        tile_regs_wait();
        pack_tile(0, cb_id_local, 0);
        tile_regs_release();
    }

    // Signal computation end
    cb_pop_front(cb_id_recv, cb_wait_tile_num);
    cb_pop_front(cb_id_local, 1);
    cb_push_back(cb_id_SE, 1);
}
}  // namespace NAMESPACE
```

Issue #113: Fixing how data is read from L1 and written to L1 (in bytes)
------------------------------------------------------------
There is a reoccurring issue where data written to L1 is not organized in same order as when packer writes data to L1, in other words our program was writing incorrectly to L1 and this caused comparison of golden and result to be incorrect even though the numbers produced were accurate they were out of order.

Once this issue was addressed and solved, our data was in stored uniform order throughout, however we were reading words from L1 using tt-exalens but misinterpreting in which order to read the bytes, and according to which format. Since this would cause too much logic and work around to develop our own model for how we read bytes from a word in L1, instead we now use tt-exalens function read_from_device which reads bytes from words in L1 correctly.

Now there is no need for bug patches made prior when reading from device in certain formats like bfp8_b or float16, everything is uniform and bytes are read in consecutive order for each format except the number of bytes varies according to size of format/datum.

Issue #105: tt-metal uses its own duplicate kernel instead of using one from the tt-llk
------------------------------------------------------------
During preparation of #99 and #100 I came across an interesting problem.

The `tt-metal` repository demonstrates three distinct patterns for handling SFPU unary operations:

1. Using its own kernel implementations:
In some cases, `tt-metal` relies on its own implementations of kernels, even if this implementation exists in tt-llk. For example, the [llk_math_eltwise_unary_sfpu_log.h](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/llk_math_eltwise_unary_sfpu_log.h) file directly uses `ckernel::sfpu::calculate_log()` from its own repository (found [here](https://github.com/tenstorrent/tt-metal/blob/24986581acac9c82e48a0e373163805e89e363a7/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/ckernel_sfpu_log.h#L69)) to perform the computation. Same applies to [llk_math_eltwise_unary_sfpu_abs.h](https://github.com/tenstorrent/tt-metal/blob/24986581acac9c82e48a0e373163805e89e363a7/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/llk_math_eltwise_unary_sfpu_abs.h) and [ckernel::sfpu::calculate_abs()](https://github.com/tenstorrent/tt-metal/blob/24986581acac9c82e48a0e373163805e89e363a7/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/ckernel_sfpu_abs.h). Same stands for `square` and trigonometry functions (`sin`, `cos` and `tanh` are present in tt-llk, while others are missing, and metal uses its own versions completely).

2. Providing an additional wrapper:
In other cases, tt-metal acts as a wrapper around existing functionality, which adds a layer of indirection (e.g. [llk_math_eltwise_unary_sfpu_typecast.h](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/llk_math_eltwise_unary_sfpu_typecast.h) which calls functions from [ckernel_sfpu_typecast.h](https://github.com/tenstorrent/tt-metal/blob/24986581acac9c82e48a0e373163805e89e363a7/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/ckernel_sfpu_typecast.h), which calls functions from tt-llk. This middle layer could be removed. Same stands for `reciprocal` and `sqrt`.

3. Calling tt-llk implementations directly
This is how I think it should be in the future, to keep all the kernels in one place (tt-llk) and call them from metal's API layer
Only a handful of tese cases exist in the metal repository at the moment, and only for grayskull, which is now de-facto defunct.


While the second pattern is not problematic in terms of functionality, it could introduce additional overhead (depending on how the code is optimized). However, the first pattern could be more concerning, as it hides changes made in tt-llk by aliasing them. What is more, tt-metal often has additional implementations (see [calculate_abs_int32](https://github.com/tenstorrent/tt-metal/blob/ff6c3218cf66e1ce26444a032f15982d00244712/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu/ckernel_sfpu_abs.h#L27) which is present in tt-metal, but not in tt-llk)

Issue #79: Fixed Hang on multiple tiles eltwise test
------------------------------------------------------------
The trisc cores responsible for unpacking, math, and packing were not running in parallel. This misalignment occurred because the cores were not loaded simultaneously. As a result, the unpacker core was lagging behind the packer and math cores by nearly 800 cycles. This discrepancy caused a race condition, where the cores were not synchronized, leading to failures in computation, particularly when reading from multiple tiles.

This issue was evident in the `test_multiple_tiles_eltwise_binary` test. The test would sometimes pass all 236 test cases, but other times, certain test cases would fail sporadically. The same test case did not always fail, and the number of failed tests varied between runs. Additionally, this issue occurred even when tests were run in isolation, with each test resetting the board and executing independently of the previous test's state.

The issue was addressed in device.py by putting all three cores into reset before loading them. Once the elf files were loaded, the cores were released from reset and started running simultaneously. Additionally, in the `trisc.cpp` file, we ensured proper synchronization by invoking the following sequence:
```cpp
run_kernel();
ckernel::tensix_sync();
```
This approach aligns with how kernels are executed in Metal, ensuring consistency and resolving the hang issue. To validate the fix, 25,000 tests were conducted on both BH and WH devices, confirming that the issue was resolved across both architectures.

Issue #77: Fix test_tilize_calculate_utilize.py synchronization bug
------------------------------------------------------------
### Problem description
Once TRISCs were synced to run at the same time in order to resolve
hangs because unpack was lagging behind pack and math TRISCs,
`test_tilize_calculate_utilize.py` started to fail because it attempted
to fuse two LLK pipeline runs. This means that in the first run we
unpack from two buffers in L1 and after math we pack to two other
buffers in L1. Then in the second run we unpack from these two buffers
that pack wrote to, however `unpacker` started to read from the buffers
before the `packer` completed writing to L1.

This worked previously because the `unpacker` was severely lagging
behind `packer` and `math` (approximately 700 cycles), and this lag
allowed for `packer` to finish writing to L1 before the `unpacker` would
start reading. Since `unpacker` is synced with `packer` we must make
`unpacker` wait for packer to finish before it begins reading from L1.

### What's changed
The bug seemed to have been fix by using a semaphore (PACK_DONE) to create communication between packer and unpacker. The packer first writes data to L1 and signals the unpacker by incrementing the semaphore (PACK_DONE = 1). The unpacker waits for the semaphore to be set to 1 before reading the data from L1.
This ensures that the unpacker does not read premature or incorrect data, preventing data race conditions.
Once the unpacker starts reading, it decrements the semaphore (PACK_DONE = 0) signalling it has started processing data.

### Bug Persists 
This might be due to the fact that math has to synchronize with unpacker. While unpacker waits for packer to write to L1 before it starts processing that data, similarly math must wait for unpacker to load data into registers before it begins operating on operands. We will try to use semaphore to synchronize math with unpacker. Hopefully this should solve the issue.

Issue #71: Rename t6_semaphore_wait_on_zero and t6_semaphore_wait_on_max for clarity
------------------------------------------------------------
Function `t6_semaphore_wait_on_zero(const uint8_t index)` is used to wait while semaphore is 0 i.e while `sem == 0`. However this is counter intuitive since "wait on zero" suggests we wait for semaphore to be 0 i.e wait while `sem != 0`.  Similarly for `t6_semaphore_wait_on_max`, the name implies waiting until the semaphore reaches its maximum value i.e while `sem!= max` or `sem < max`, but it in fact waits while `sem == max` which could lead to confusion.

```cpp
template <uint WaitRes>
inline void t6_semaphore_wait_on_max(const uint8_t index)
{
    TTI_SEMWAIT(WaitRes, semaphore::t6_sem(index), p_stall::STALL_ON_MAX);
}

template <uint WaitRes>
inline void t6_semaphore_wait_on_zero(const uint8_t index)
{
    TTI_SEMWAIT(WaitRes, semaphore::t6_sem(index), p_stall::STALL_ON_ZERO);
}
```

Notice that our `wait_sem_cond` for `TTI_SEMWAIT` is `STALL_ON_ZERO` and `STALL_ON_MAX`, there respective values are shown below.
```cpp
//ckernel_instr_params.h
constexpr static uint STALL_ON_ZERO = 0x1;
constexpr static uint STALL_ON_MAX  = 0x2;
```
However, we can see that `SEMWAIT` will acknowledge that passing argument wait_sem_cond is 1 i.e `STALL_ON_ZERO` then it waits while semaphore is 0 and when `wait_sem_cond` is 2 i.e `STALL_ON_MAX` it waits while semaphore is max.
```yaml
SEMWAIT:
    instrn_type: LOCAL_CREGS
    ex_resource: SYNC
    op_binary: 0xa6
    src_mask: 0x0
    fcov:
    arguments:
        - name: wait_sem_cond
          start_bit: 0
          field_type: DEC
          description: "Wait on semaphore. 1 - wait while semaphore is 0, 2 - wait while semaphore is max"
          fcov_point_bins:
            bins: [ {name: "WaitWhileZero",    value: "0x1"},
                    {name: "WaitWhileMax",  value: "0x2"}]
```

Renaming the functions to `t6_semaphore_wait_while_zero` and `t6_semaphore_wait_while_max` would more accurately reflect its purpose. It clearly indicates that the function is waiting while the semaphore value is zero, not until it reaches zero. This change would make the code more intuitive for developers who read the function name and try to understand its behavior at a glance.

Issue #61: [Blackhole]: Pack untilize improvements
------------------------------------------------------------
Pack untilize kernel needed to be re-written for Blackhole, due to Blackhole packer limitations:

Only one Dest offset register for the packer instance (but we can use strided mode of 16 rows apart)
PACR instructions can only write L1 contiguous output.
These L1 Offset registers, are now set per PACR Context, not per rows output from PACR instruction (i.e PACK_INTF_SEL)

```
THCON_SEC0_REG1_L1_Dest_addr_ADDR32 -> Context 0
THCON_SEC0_REG8_L1_Dest_addr_ADDR32 -> Context 1
THCON_SEC1_REG1_L1_Dest_addr_ADDR32 -> Context 2
THCON_SEC1_REG8_L1_Dest_addr_ADDR32 -> Context 3
```

Here is the current algorithm:

![Image](https://github.com/user-attachments/assets/2368edba-a9ad-4e5c-9952-df51b1b474a6)

For wormhole b0, 8x16 rows can be output per tile (before incrementing counters), while for blackhole with the above implementation, only 2x16 can. Since pack_untilize feature needs to be fused with other operations, we cannot change the unpacker to use a different unpacking scheme (Fastest untilize scheme would be T0F0, T0F1, T1F0, T1F1, T0F2, T0F3, T1F2, T1,F3)). However, we should use that implementation if the use case is doing a pack_untilize for a block of tiles (i.e not fused).

One major optimization, is to enable the use of contexts.
```
THCON_SEC0_REG1_L1_Dest_addr_ADDR32 = CNTX0 = tile_offset (Configured for top faces)
THCON_SEC0_REG8_L1_Dest_addr_ADDR32 = CNTX2 = tile_offset + block_ct_dim * TILE_C_DIM *datum_size ((Configured for bottom faces)
```
Then algorithm can do 4x16 rows, will change to something like this:

```
CH0 x_stride = number of bytes per datum (2 bytes for Float16b default)
CH0 y_stride = FACE_C_DIM*x_stride
CH0 z_stride = FACE_R_DIM*y_stride (1x16x16 datums default, already done in pack_hw_config)
CH0 w_stride = 4*z_stride (32x32 by default)
PACK_INTF_SELECT_0 = 0b0011 (read 2 rows from dest)
PACK_INTF_SELECT_1 = 0b1100 (read 2 rows from dest)
Dest Mode = DST_STRIDED_MODE (Each row read from dest is 16 apart)

    for face_r_dim (16 by default):
        for block_ct_dim (2 in this example) {
            PACK(CNTX0, PACK_INTF_SELECT_0); (Reads 2 rows, writes them contigous in L1, dest rows = 0,16, 1, 17, 2, 18 .....)
            PACK(CNTX1, PACK_INTF_SELECT_1); (Reads 2 rows, writes them contigous in L1, dest rows = 32,48, 33, 49,  .....)
            W_CNT += 1 (Jumps to next tile)
        }
        Y_CNT+=1
    }
```
The problem here is if contexts are used, initial hardware configuration for data formats, tile sizes, etc, all need to also be programmed.

Issue #55: [Blackhole] CFGSHIFTMASK instruction
------------------------------------------------------------
Blackhole has new CFGSHIFTMASK that can update addresses for the unpacker instructions inside the mop/replay buffers.

Instead of updating addresses in this method:

```
  TTI_RDCFG(p_gpr_unpack::TMP0, THCON_SEC0_REG3_Base_address_ADDR32);
  TTI_ADDDMAREG(0, p_gpr_unpack::TMP0, p_gpr_unpack::TMP0, p_gpr_unpack::TILE_SIZE_A);
  TTI_STALLWAIT(p_stall::STALL_CFG, p_stall::THCON);
  TTI_WRCFG(p_gpr_unpack::TMP0,0,THCON_SEC0_REG3_Base_address_ADDR32);
```

Using the CFGSHIFTMASK instruction, it could be done like this:

```
TTI_CFGSHIFTMASK(1, 0b011, 32 - 1, 0, 0b11, THCON_SEC0_REG3_Base_address_ADDR32); // THCON_SEC0_REG3_Base_address_ADDR32 =  THCON_SEC0_REG3_Base_address_ADDR32 +  SCRATCH_SEC0_val_ADDR32 
```

as long as the scratch buffer is correctly populated:

```
TTI_WRCFG(p_gpr_unpack::TILE_SIZE_A, 0, SCRATCH_SEC0_val_ADDR32);
TTI_NOP;
```

If an operation is unpacker bound, then using the CFGSHIFTMASK should increase performance.

Issue #20: [Code improvement] Stop using `using namespace` in header files
------------------------------------------------------------
### Description
Using using namespace in header files is generally considered bad practice because it can lead to namespace pollution and unexpected name conflicts. When you include a header file that uses using namespace, it brings all the names from the specified namespace into the global namespace of any file that includes it. This can cause name collisions and make the code harder to understand and maintain. At the moment of writing this issue, we have 183 using directives in 149 files. Since this is a header-only project, I'd say we have this issue in almost 60% of files.

### Why This Is a Problem
**Namespace Pollution:** All names from the "used" namespaces are brought into the global namespace, which can lead to unintended name clashes.
**Code Maintainability:** It becomes difficult to track where certain names are coming from, making the code harder to read and maintain.
**Potential Conflicts:** If another header file or source file includes the same header and defines names that conflict with those in the namespace that was included using `using namespace` directive, it can lead to compilation errors or unexpected behavior.

### Solution
**Remove using namespace directives:** Remove the using namespace directives from all of the header files and fully qualify the names from namespaces where they are used.
